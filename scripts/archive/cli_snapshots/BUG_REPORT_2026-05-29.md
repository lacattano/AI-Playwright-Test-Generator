### BUG-1: Screen Clipping — Output Truncated in Subprocess Pipe

**Severity:** High  
**File:** `cli/menu_renderer.py` → `_capture_output()`  
**Snapshot:** `menu_navigation.md` Step 1, `screen_clipping.md`

**Symptom:** When the CLI runs via subprocess pipe (no TTY), the captured output is severely truncated. Step 1 shows only:
```
┌──────────────────────────────────────────────────────────────────────────┐
│  AI Playwright
```
The full header (title, subtitle, menu items, shortcut bar) is missing.

**Root Cause:** `_capture_output()` uses `io.StringIO()` to capture stdout. However, `sys.stdout.write()` calls with `flush=True` write to the **original** stdout when running in a subprocess pipe context, not the captured StringIO. The capture function sets `sys.stdout = StringIO()` but the rendering functions use `sys.stdout.write(..., flush=True)` which may bypass the capture in pipe mode.

**Impact:** Debug scripts cannot capture full screen output for analysis. Users running CLI in non-interactive contexts (CI, logging) get truncated output.

**Fix:** Ensure `_capture_output()` captures all `sys.stdout.write()` calls. The issue is likely that `flush=True` on a StringIO doesn't propagate, or the ANSI clear screen (`\033[H\033[J`) clears the captured buffer. Consider:
1. Not clearing screen when stdout is not a TTY
2. Using a different capture mechanism that doesn't rely on StringIO

### BUG-2: ANSI Escape Codes Leaking into Output

**Severity:** Medium  
**File:** `cli/retro_ui.py` → `clear_screen()`  
**Snapshot:** `menu_navigation.md` Step 1, Step 2, Step 3

**Symptom:** Raw ANSI escape sequences (`[H[J`) appear in the captured output:
```
[H[J┌──────────────────────────────────────────────────────────────────────────┐
```

**Root Cause:** `clear_screen()` writes `\033[H\033[J` unconditionally. When stdout is not a TTY (subprocess pipe), these escape codes are written as literal text instead of being interpreted. The `_green()` function in `color.py` has a `sys.stdout.isatty()` check, but `clear_screen()` does not.

**Impact:** 
- Debug snapshots contain noise
- CLI output logged to files contains escape codes
- Non-ANSI terminals show garbage characters

**Fix:** Add TTY check to `clear_screen()`:
```python
def clear_screen() -> None:
    if sys.stdout.isatty():
        sys.stdout.write("\033[H\033[J")
        sys.stdout.flush()
```

Same fix needed for `hide_cursor()`, `show_cursor()`, `move_cursor()`.

### BUG-3: Menu Navigation State Not Persisting Between Screens

**Severity:** High  
**File:** `cli/menu_renderer.py` → `render_main_menu()`  
**Snapshot:** `menu_navigation.md` Step 2, Step 3

**Symptom:** After pressing Down arrow, the snapshot shows two screens:
1. First screen shows `> Configure LLM` (old selection)
2. After clear, shows `> Enter User Story` (new selection)

The old screen content appears in the capture before the clear. This means the selection index is being updated, but the screen clear happens **after** the old content is printed, not before.

**Root Cause:** `render_main_menu()` calls `clear_screen()` at the top, but in subprocess pipe mode, the previous screen's output is already captured. The clear ANSI codes don't erase the captured text, they just append escape sequences.

**Impact:** Each menu redraw appends to the captured output rather than replacing it. This makes the output hard to read in logs and causes the debug snapshots to contain multiple screen states.

**Fix:** When not running in a TTY, use a separator instead of clear:
```python
def clear_screen() -> None:
    if sys.stdout.isatty():
        sys.stdout.write("\033[H\033[J")
    else:
        sys.stdout.write("\n" + "=" * 80 + "\n")
    sys.stdout.flush()
```

### BUG-4: Shortcut Bar Truncation Shows Ellipsis Incorrectly

**Severity:** Low  
**File:** `cli/retro_ui.py` → `render_shortcut_bar()`  
**Snapshot:** `menu_navigation.md` Line 33

**Symptom:** Shortcut bar shows:
```
[1]Configure LLM  [2]Enter User Story  [3]Save & Exit  [4]Quit  [Q]Q  [...]
```

The `[Q]Q` entry is duplicated/truncated. The shortcut bar has `[Q]Quit` but it's being cut off and shown as `[Q]Q`, then `...` is appended.

**Root Cause:** `render_shortcut_bar()` at line 261-262 truncates when `len(shortcut_text) > inner`. The truncation cuts mid-character, showing `[Q]Q` instead of `[Q]Quit`. The `...` is appended but the cutoff point is ugly.

**Impact:** Visual only — shortcuts still work. But looks unprofessional.

**Fix:** Truncate at word boundaries:
```python
if len(shortcut_text) > inner:
    # Truncate at last "  " separator that fits
    truncated = shortcut_text[: inner - 3]
    last_space = truncated.rfind("  ")
    if last_space > 0:
        truncated = truncated[:last_space]
    shortcut_text = truncated + "..."
```

### BUG-8: `_read_key()` Blocks Indefinitely in Git Bash (CRITICAL)

**Severity:** Critical  
**File:** `cli/menu_renderer.py` → `_read_key()`  
**Diagnosed by:** `scripts/debug/debug_cli.py`

**Symptom:** `_read_key()` blocks indefinitely in Git Bash (MINGW64). The debug script confirms:
- `_read_key()` does not return after 3 seconds (thread still alive)
- `select.select()` fails with `WinError 10093` (WSAStartup not called)
- `termios`/`tty` not available on Windows
- `sys.stdin.readline()` blocks waiting for Enter in Git Bash

**Root Cause:** `_read_key()` uses `sys.stdin.read(1)` or similar low-level stdin read that blocks in Git Bash. Git Bash on Windows provides a pseudo-TTY that doesn't support:
1. `select()` on stdin (Windows socket error)
2. `termios/tty` module (Unix-only)
3. Non-blocking character reads (requires Enter keypress to flush)

**Impact:** **The CLI is fundamentally broken in Git Bash.** Users cannot use arrow keys or single-key navigation. The CLI only works if users type a key AND press Enter, which defeats the purpose of an interactive menu.

**Evidence from debug_cli.py:**
```
_running_in_git_bash(): True
_read_key() is BLOCKING after 3 seconds (thread still alive)
This confirms the bug: _read_key() does not return in Git Bash
select failed: [WinError 10093] Either the application has not called WSAStartup
termios/tty not available
PROBLEM: sys.stdin.readline() blocks waiting for Enter in Git Bash.
```

**Fix Options:**
1. **Use `msvcrt.getch()` for native Windows consoles** — works in cmd.exe/PowerShell
2. **Detect Git Bash and fall back to line-based input** — `input()` with parsing
3. **Use `curses` library** — not available in Git Bash on Windows either
4. **Use `getch()` from third-party package** — `getch` package or `readchar`
5. **Best approach:** Detect environment and use appropriate input method:
   - Native Windows console: `msvcrt.getch()`
   - Git Bash / WSL: line-based input with Enter key
